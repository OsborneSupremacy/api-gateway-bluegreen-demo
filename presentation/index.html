What is Blue/Green Deployment?

🔵 Blue (Current Production)

  • Live, serving 100% of real traffic
  • Known-good state
  • Your rollback target

🟢 Green (Next Version)

  • Deployed alongside Blue
  • Isolated — real traffic not yet routed here
  • Available for validation & testing
The big idea: promote Green to Blue via a control-plane switch — no redeployment, no traffic spike, no crossed fingers. If something looks wrong, don't promote.

Why Blue/Green?

🛡️ Risk Reduction

Validate in the real environment before any customer is affected. Rollback is instant — just don't promote.

⚡ Zero Downtime

Promotion is a config change (Lambda alias pointer / stage variable update), not a redeployment. Takes milliseconds.

🔍 Production Parity

Green runs in the same account, same region, same VPC — not a staging clone that lies to you.

↩️ Safe Rollback

Blue stays warm and intact until you explicitly retire it. Rollback = no action needed.

🧪 Smoke & Canary Friendly

Run automated smoke tests against the green URL before any promotion decision.

🔑 Control Plane, not Data Plane

Promotion touches config — not code. Minimises blast radius of the promotion itself.

The AWS Building Blocks

🔶 AWS Lambda

  • Versions — immutable snapshots of code + config
  • Aliases — named pointers to a version (blue, green, live)
  • Alias ARN is stable; consumers don't change when you re-point it

🟣 Amazon API Gateway

  • Stages — named deployments of an API (blue, green)
  • Stage Variables — key/value config injected at runtime
  • Each stage gets its own invoke URL

How they fit together

A stage variable (e.g. lambdaAlias) tells the integration which Lambda alias to call. Flip the variable → promotion complete.

# Stage variable example lambdaAlias = blue ← production lambdaAlias = green ← pre-promotion # Integration ARN (parameterised) arn:aws:lambda:us-east-1:123:function: MyFunction:${stageVariables.lambdaAlias}

No code change. No redeployment. Just a stage-variable update.

Lambda Versions & Aliases

Versions (immutable)

  • Published from $LATEST
  • Numbered sequentially: 1, 2, 3 …
  • Code & config are frozen at publish time
  • Can't be modified after creation

Aliases (mutable pointers)

  • Human-readable name → version number
  • Stable ARN regardless of which version it points to
  • Can split traffic between two versions (weighted alias)
  • In our model: blue = live, green = candidate

Anatomy of a promotion

# Before promotion blue alias → version 7 (live) green alias → version 8 (candidate) # Smoke tests pass ✅ # Promotion: update blue alias aws lambda update-alias \ --function-name MyFunction \ --name blue \ --function-version 8 # After promotion blue alias → version 8 (live ✅) green alias → version 8 (still points here) # Cleanup (later) # Retire version 7 when confident

The alias ARN seen by API Gateway never changes. Zero client impact.

API Gateway Stages

What a Stage gives you

  • A unique invoke URL:
    https://<id>.execute-api.<region>.amazonaws.com/blue
  • Independent throttling, caching & logging settings
  • Stage variables — runtime config injected into integrations
  • Deployment history — roll back a stage independently

Stage Variables for integration routing

  • Passed to Lambda as part of the event
  • Used in the integration URI to parameterise the alias
  • Update via console, CLI, or IaC

Stage variable — routing example

# blue stage lambdaAlias = blue invokeUrl = /blue # green stage (separate URL, separate config) lambdaAlias = green invokeUrl = /green # Lambda integration URI (shared by both stages) arn:aws:apigateway:us-east-1:lambda:path/ 2015-03-31/functions/ arn:aws:lambda:us-east-1:123456789: function:MyFunction: ${stageVariables.lambdaAlias}/invocations
Green gets a real URL you can test against in production. Blue is untouched. Promotion = update the blue stage variable.

1. Deploy Green

🔍

✅

  • Re-deploy and re-verify

Step 1: Deploy Green

① Build & publish Lambda version
aws lambda update-function-code \ --function-name MyFunction \ --zip-file fileb://package.zip aws lambda publish-version \ --function-name MyFunction # → returns VersionNumber: 8
② Point green alias to new version
aws lambda update-alias \ --function-name MyFunction \ --name green \ --function-version 8
③ Create/update API Gateway green stage
# Create a deployment for the green stage aws apigateway create-deployment \ --rest-api-id abc123 \ --stage-name green \ --variables lambdaAlias=green

Step 2: Verify

🧪 Automated Smoke Tests

  • Point your test suite at the green URL
  • Exercise critical paths / contracts
  • Assert on status codes, response shapes, latency
  • Run in CI/CD pipeline before promotion gate

👁️ Observability Checks

  • CloudWatch Lambda metrics (errors, duration, throttles)
  • API Gateway access logs (4xx / 5xx rates)
  • X-Ray traces (latency anomalies)
  • CloudWatch Alarms as promotion gates

🙋 Manual Sign-off (optional)

  • Share the green URL with QA / stakeholders
  • They verify against production data, real infrastructure
  • Approval gate in your pipeline (CodePipeline approval, GitHub environment protection)
Key insight: Green runs in production — same DB, same caches, same downstream services. No "it worked in staging" surprises.

Step 3a: Promote

Verification passed — time to make Green the new Blue.

① Update blue Lambda alias
aws lambda update-alias \ --function-name MyFunction \ --name blue \ --function-version 8 # blue now → version 8 ✅
② (Optional) Re-deploy blue API stage

Only needed if stage-level config (throttling, caching) changed. If only the Lambda alias changed, no API Gateway update is required.

Before & After Promotion

AliasBeforeAfter
blueversion 7version 8 ✅
greenversion 8version 8 (unchanged)
Zero downtime: The blue alias ARN never changed. API Gateway keeps calling the same ARN. Lambda silently serves version 8 instead of 7. In-flight requests on version 7 complete naturally.

Cleanup (when confident)

  • Retire or delete the green API stage
  • Delete old Lambda version 7 (or let it age out)

Step 3b: Don't Promote — Rollback

Verification failed (or you changed your mind). Blue is still live and unaffected.

What rollback means in this model

  • Do nothing — Blue was never touched
  • Delete or re-deploy the green stage with a fixed version
  • No hotfix, no emergency redeploy, no phone calls at 2 AM

Fix & Re-verify cycle

  • Fix the issue in code
  • Publish a new version (9, 10, …)
  • Update green alias → new version
  • Re-run verification
  • Repeat until it passes
The safety net: Because promotion is a control-plane operation — not a traffic reroute or load-balancer drain — the blast radius of a bad deployment is limited to whoever is testing the green URL. Zero customers are affected, zero SLA impact.

Observability & Promotion Gates

📊 CloudWatch Metrics

  • Lambda: Errors, Duration, Throttles, ConcurrentExecutions
  • API GW: 4XXError, 5XXError, Latency
  • Scope metrics to the green alias / stage

🔍 X-Ray Tracing

  • Enable active tracing on the green alias
  • Identify cold-start regressions
  • Trace end-to-end latency through downstream services

📋 Structured Logs

  • Emit version in every log line
  • Filter CloudWatch Logs Insights by version
  • Easier post-promotion comparison

🚦 Promotion Gate Pattern

# CodePipeline / GitHub Actions gate example (pseudo-code) 1. Deploy Green → trigger pipeline 2. Run smoke tests against /green URL 3. Wait N minutes → check CloudWatch alarm state - Alarm: GreenErrorRate > 1% → ABORT promotion, alert team - Alarm: GreenP99Latency > 2s → ABORT promotion, alert team 4. (Optional) Manual approval step 5. All clear → promote (update blue alias)

IaC Considerations

🏗️ What to manage in IaC (Terraform / CDK / SAM)

  • Lambda function definition (not version — let pipeline publish)
  • Lambda aliases (blue, green) with placeholder versions
  • API Gateway REST API & shared resources
  • Both stages (blue, green) with stage-variable schema
  • IAM roles, CloudWatch alarms, X-Ray config

⚙️ What to manage in your pipeline

  • Publishing Lambda versions (publish-version)
  • Updating alias version pointers
  • Running smoke tests
  • Promotion (alias flip)
Tip: Separate the infrastructure lifecycle (IaC) from the deployment lifecycle (pipeline). IaC creates the slots; the pipeline fills them.

Common Pitfalls

❌ Stateful schema migrations

If your Lambda writes to a DB, a green version with a breaking schema change will affect the shared database used by blue. Use backward-compatible migrations first.

❌ Shared caches / event sources

SQS queues, Kinesis streams, and DynamoDB Streams are not stage-aware. Messages processed by green may have side effects visible to blue consumers. Plan accordingly.

❌ Forgetting Lambda resource-based policies

API Gateway must have invoke permission on both aliases (blue and green). Add permissions at alias level, not at function level.

❌ Treating $LATEST as an alias

$LATEST is mutable and not suitable for production routing. Always publish a version and target a named alias.

Key Takeaways

🔵🟢

Two slots, one function

Lambda aliases and API Gateway stages give you two independent traffic paths on a single function with no extra cost.
⚡

Promotion is a config change

An alias pointer update is milliseconds of latency, not a redeployment. Zero-downtime is structural, not heroic.
🛡️

Rollback is "do nothing"

Blue never changes until you explicitly promote. If verification fails, Blue keeps serving customers without incident.
🔍

Production parity

Green runs in the same account, same region, same services. No staging-vs-prod surprises at promotion time.
🚦

Gate on signals

Automate promotion gates: smoke tests + CloudWatch alarms = confident promotion without manual guesswork.
🏗️

Separate infra from deploy

IaC owns the slots. Your pipeline fills them. Keeps both concerns clean and independently evolvable.
